# 小熊猫 · 开发计划与可行性分析

> 基于 Gemini 讨论整理 · 技术落地与优先级补充

## 一、功能可行性总览

| 功能 | 可行性 | 难点 | 更现实的做法 |
|------|--------|------|--------------|
| **龙虾螯·自动订票** | 中 | 12306/携程反爬、RPA 风控 | 意图识别 + Deep Link 预填跳转，用户手动支付 |
| **碎片化捕获·语音/截图** | 极高 | 冲突消解算法 | 多模态 API 解析 + 本地冲突检测函数 |
| **龙虾壳·智能挡箭牌** | 较高 | 依赖日历/邮件 API | 企业生态（飞书/钉钉）易实现；个人需系统日历读写 |
| **龙虾钳·WBS 拆解** | 高 | 无 | 已有 parse-tasks，扩展拆解逻辑即可 |
| **龙虾须·情境准备** | 较高 | 会议纪要来源 | 需接入会议/文档 API |
| **财务/支出环形图** | 中 | 微信/支付宝 API 权限高 | 引导用户上传账单 CSV，解析后展示 |
| **消费代理·自主搜寻** | 较高 | 需电商搜索 API | 调用淘宝/京东开放能力或爬虫（需合规） |
| **找房/砍价话术** | 较高 | 房产平台 API | 监控房源 + 话术生成，不发敏感信息 |

## 二、技术深水区补充

### 2.1 编排层（Agent Orchestration）

单次 Prompt 无法完成复杂任务，需要引入工作流框架：

- **LangGraph** / **CrewAI**：支持「思考 → 计划 → 行动 → 观察」的 ReAct 循环
- **Vercel AI SDK**：`streamText` + `tool` 实现 Function Calling

### 2.2 前端通信

- 用 **SSE** / **WebSocket** 流式推送 Agent 执行步骤
- 避免死板 Loading 圈，每步实时显示：「正在搜索车票…」「发现两班高铁…」「正在生成行程单…」

### 2.3 记忆隔离

| 类型 | 存储 | 用途 |
|------|------|------|
| 短期记忆 | Redis / 内存 | 当前会话连续对话 |
| 长期偏好 | Vector DB | 用户习惯（如：从不坐 8 点前飞机） |
| 日程数据 | Turso | 项目、任务、冲突检测 |

## 三、技术原理速览（ReAct 模式）

```
Thought（思考）→ Action（行动）→ Observation（观察）→ 循环至完成
```

- **Thought**：用户想订票，我需要先查周五日程
- **Action**：执行查日程
- **Observation**：周五下午 3 点有空
- **Thought**：现在要查 12306
- **Action**：生成 Deep Link 跳转
- **Observation**：用户已跳转，任务完成

## 五、推荐开发顺序（与 PRD 一致）

```
1. 语音输入增强 + 日历视图     ← 先做强感知与可视化

```

## 六、避坑指南

### 6.1 隐私与权限

- **初期不做全自动扣款**：AI 选好方案后，用户手动点击支付
- **数据隔离**：敏感日程建议 TEE 或本地加密
- **权限确认**：发邮件、删文件、支付前弹「允许/拒绝」

### 6.2 普通人可用

- **消除黑窗口**：图形化界面，一键安装
- **内置保险箱**：沙箱环境，AI 只读指定 Workspace
- **傻瓜式连接**：扫码即用，不写配置文件
- **预设技能卡**：一键周报、清理下载文件夹等，减少空白感

### 6.3 2026 技术杀手锏

- **MCP (Model Context Protocol)**：Agent 可直接读 Google Calendar、本地文件、Slack，无需逐一手写 API
- **小程序端**：微信 `voip-room` 或语音接口，可实现息屏语音唤醒
