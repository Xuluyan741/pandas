## 一、逐轮聊天纪要（按时间顺序梳理）

> 下文不是逐字逐句录音稿，而是在不改变含义的前提下按轮次「忠实还原」对话内容，便于阅读与存档。

1. **用户问题：终端 vs 网站，能力是否缺失？**  
   - 背景：Nanobot、OpenClaw 等多为终端/本地运行；小熊猫是 Web 端。  
   - 关切：自己不是开发者，担心小熊猫在网站运行会不会导致能力不如终端系 Agent（Nanobot、OpenClaw、NanoClaw 等），特别是「随用随下」这类高级特性。

2. **Agent 说明：形态差异 ≠ 能力被阉割**  
   - 讲清：  
     - Nanobot / NanoClaw / Claude Code：**开发者、终端、单用户、自建环境**，擅长 Shell、多通道、本地资源调用。  
     - 小熊猫：**Web、面向普通用户、多用户账号体系**，聚焦「日程/任务/冲突/长期目标」，不直接暴露终端或 Shell。  
   - 结论：小熊猫在自己定义的产品边界内（智能日程管家 + 冲突消解 + 长期目标 + 轻量 Token），网站形态不会导致能力缺失，只是**有意不做** Shell、多通道、本地 vLLM 等系统级能力。

3. **用户追问：MCP 可以帮用户做什么？**  
   - 关切：MCP 听起来很重要，想知道对最终用户到底意味着哪些「可感知的功能」。  
   - Agent 回答：  
     - MCP 就是一种「把外部能力当工具挂给 AI」的协议。  
     - 对用户而言：只要在小熊猫这边配置好 MCP 服务器，**就能用一句话调用那台 MCP 服务器提供的所有工具**——比如读/写文件、联网搜索、查数据库、Git/GitHub 操作、日历/邮件、内部 API 等。  
     - 小熊猫负责：理解意图 → 选工具 → 调 MCP → 把技术细节包装成自然语言回复。

4. **用户指出：OpenClaw 有 5700 个技能**  
   - 关切：OpenClaw 通过 ClawHub 集成了 5700+ 技能，为何感觉小熊猫没那么「丰富」？  
   - Agent 说明：  
     - 那 5700+ 技能来自 **ClawHub**；OpenClaw 与小熊猫接的是**同一个 ClawHub**。  
     - 差异在于：  
       - OpenClaw：`clawhub install` 安装到本地 workspace，可以真正执行技能脚本。  
       - 小熊猫：通过 ClawHub API **语义搜索 + 拉 SKILL.md**，以「即用即删」方式把说明喂给模型，不做本地安装或执行。  
     - 所以在「能从哪个池子找技能」上，小熊猫和 OpenClaw 是一致的；差别是：  
       - 前者更多是「参考说明，生成建议/计划」；  
       - OpenClaw 可以「执行」更多系统层面的动作。

5. **用户进一步质疑：主页实际体验里完全看不到技能/ClawHub 的影子**  
   - 关切：虽然文档和架构都说支持 ClawHub / MCP / 即用即删，但实际在主页和小熊猫对话时，「感觉什么都没发生」，没有任何关于技能的提示或可视化。  
   - Agent 排查：  
     - 后端：  
       - `/api/agent/chat` 中已经通过 `discoverClawHubSkillsForTask` 对当前输入做 ClawHub 语义搜索，并在 system prompt 里增加一段「本轮参考的社区技能」文本；  
       - 但 **没有**通过 SSE 把「这轮参考了哪些社区技能」传回前端。  
     - 前端：  
       - `PandaChat` 仅消费 `thought` / `reply` / `error` 三种事件；  
       - 消息结构里没有 `communitySkills` 字段，也没有任何 UI 区块用来展示「参考了哪些技能」。  
   - 诊断结论：  
     - 后端确实在「静默」使用 ClawHub 结果，但**没有可见化到前端**，所以用户感觉「毫无体现」是合理的。

6. **用户请求：帮忙把本窗口的讨论与结论保存为文档**  
   - 目标：便于后续复盘和与他人分享，对 MCP / ClawHub / 小熊猫 vs 终端系 Agent 的认知保持一致。  
   - 本文即为「讨论记录 3」。

## 二、关键结论（给产品 / 技术 / 合伙人看的高层摘要）

1. **形态结论：Web 端小熊猫 ≠ 终端削弱版**  
   - 小熊猫有意放弃的是 Shell、多通道、本地 vLLM 等系统级能力，聚焦在**智能日程管家**（解析 → 冲突检测 → Deep Link → 长期目标）。  
   - 在这条主线上，网站形态不会导致能力被削弱，反而利于多用户、账号体系、权限与计费等设计。

2. **MCP 结论：是「外部能力网关」，而不是某个单一功能**  
   - 一旦配置 MCP 服务器，用户就可以通过对话使用这台服务器上的 **所有工具**：  
     - 文件系统 / 搜索 / 数据库 / Git / 日历 / 邮件 / 内部 API / 自定义业务能力等。  
   - 小熊猫当前已经有：  
     - `mcp_call` 技能；  
     - `/api/agent/mcp/tools/for-intent`（按意图只挂载部分工具）；  
     - `/api/agent/mcp/chat`（一站式 MCP 对话）。

3. **ClawHub / 5700+ 技能结论：池子相同，用法与执行力度不同**  
   - OpenClaw 与小熊猫共享的「技能池」都是 ClawHub 5700+ 社区技能。  
   - 当前差异：  
     - OpenClaw：偏 **安装 + 可执行脚本**，执行能力更强，但也更偏开发者和自建环境。  
     - 小熊猫：偏 **语义搜索 + 即用即删地注入 SKILL.md**，主要用于增强「理解与建议」，执行层面更保守。

4. **体验结论：原实现中，ClawHub 的存在几乎是「隐形」的**  
   - 后端利用了 ClawHub 结果构造更强 Prompt，但没有前端可视化。  
   - 对非技术用户来说，这等价于「感觉没这玩意」——需要通过状态面板和回复气泡给出明确反馈。

5. **安全与成本结论：继续坚持「即用即删 + 按意图挂载 + 不在 Next 中执行任意代码」**  
   - 可执行层面的技能建议通过 MCP Server 或独立 Worker 实现；  
   - Next.js Web 进程只担当「路由 + Prompt 组装 + 工具调用入口」，不做脚本执行容器，避免安全风险。

## 三、当前配置重点（与本次讨论相关的 env / 模块）

### 1. ClawHub 相关

- `CLAWHUB_API_BASE`  
  - 默认：`https://clawhub.ai`  
  - 设为 `0` 时关闭 ClawHub 对接（讨论中强调：若想体验技能市场能力，**不要**设为 0）。
- 相关代码：  
  - `src/lib/skills/clawhub.ts`：ClawHub API 客户端（搜索、按 slug 获取详情、拉 SKILL.md）。  
  - `src/app/api/skills/store/*`：对外暴露搜索与详情接口。  
  - `src/app/api/agent/chat/route.ts`：在主对话里按当前输入从 ClawHub 找 top2 技能，并以「即用即删」方式注入。

### 2. MCP 相关

- 环境变量（`.env.local.example` 中已示例）：  
  - `MCP_SERVER_URL`：HTTP MCP 服务器地址。  
  - `MCP_SERVER_HEADERS`：JSON 格式的请求头（如 `{"Authorization":"Bearer xxx"}`）。
- 代码模块：  
  - `src/lib/mcp-client.ts`：最小 MCP HTTP 客户端，支持 `tools/list`、`tools/call`。  
  - `src/app/api/agent/mcp/tools/for-intent`：按意图只返回本轮应挂载的 MCP 工具。  
  - `src/app/api/agent/mcp/chat`：探测 → 筛选 → 挂载 → 工具调用 → 再补全的一站式对话。  
  - `src/lib/skills/registry.ts` 中的 `mcp_call` 技能：统一入口，供主 Agent 调用 MCP 工具。

### 3. 主页对话 / PandaChat 相关

- 后端（`src/app/api/agent/chat/route.ts`）：  
  - CLawHub：`discoverClawHubSkillsForTask(text, 2, 1800)` 搜索当轮匹配技能。  
  - 记忆：`loadMemoryForPrompt(userId)` 合并用户记忆。  
  - SSE 事件：`thought` / `reply` / `error`（并在新版本中增加了 `skills_used` 事件与 `communitySkills` 字段，用于前端显式展示）。  
- 前端（`src/components/chat/PandaChat.tsx`）：  
  - 解析 SSE 事件，维护 `messages` 与 `agentLogs`。  
  - 渲染助手回复、冲突检测卡片、长期目标卡片、Deep Link 执行卡片等。  
  - 新增：对 `skills_used` 与 `communitySkills` 的解析与展示（状态面板 + 消息气泡下方的小字提示）。

## 四、后续测试清单（围绕本次讨论涉及的功能）

> 可在 `docs/功能测试清单.md` 中补充或勾选对应项。这里是与本次话题直接相关的测试角度。

### A. ClawHub / 社区技能相关

- **A1：默认启用 ClawHub 时，主页对话能否看到「本轮参考的社区技能」提示**  
  - 前提：`.env.local` 中 **不**设 `CLAWHUB_API_BASE=0`。  
  - 步骤：  
    1. 启动 dev 环境，打开主页。  
    2. 输入与某些常见技能高度相关的语句（如「帮我生成一份周报草稿」）。  
    3. 观察右侧/底部状态面板是否出现「本轮参考了 x 个社区技能：…」。  
    4. 查看对应回复气泡下方是否有「本轮参考了 x 个社区技能」的小字提示，并检查链接能否正确打开 ClawHub 技能页面。

- **A2：关闭 ClawHub 时行为是否优雅退化**  
  - 前提：`.env.local` 中设置 `CLAWHUB_API_BASE=0`。  
  - 预期：  
    - 对话不再展示任何「社区技能」相关提示；  
    - 对话仍能完成解析/冲突检测/长期目标等主流程，不报错。

### B. MCP 相关

- **B1：`/api/agent/mcp/tools/for-intent` 可以按意图筛工具**  
  - 配置：`MCP_SERVER_URL` 指向一台提供若干工具（如 `read_file`、`search_web`）的 MCP 服务器。  
  - 测试：  
    - 用「读取文件内容」类的 `text` 调用接口，预期返回包含 `read_file`。  
    - 用纯闲聊语句调用，预期返回空工具列表。

- **B2：`/api/agent/mcp/chat` 能完成一轮「发现 → 挂载 → 调用 → 回答」闭环**  
  - 给一个需要某个 MCP 工具的问题（如「帮我查看项目根目录下的 README 内容」）。  
  - 预期返回自然语言回答，并可在日志中观察到 `tools/list` 与 `tools/call` 的调用。

### C. 主页对话体验相关

- **C1：非技术用户视角，是否能明显感知「小熊猫在调用外部能力」**  
  - 包括：  
    - 状态面板中的 thought 文案是否清晰友好（如「正在理解…」「正在检查冲突…」「本轮参考了 x 个社区技能：…」）。  
    - 消息气泡下相关提示是否不会造成视觉噪音（字号、颜色、位置合适）。

- **C2：在 ClawHub / MCP 不可用时的降级体验**  
  - 模拟网络失败或配置错误；  
  - 预期主流程仍然可用，只是少了某些增强能力；  
  - 错误提示对用户友好，不出现技术栈细节。

## 五、给未来自己的简短建议

- 若后续要向「更像 OpenClaw」演进（多通道、可执行技能、更强自动化），优先路线是：  
  1. **所有可执行技能以 MCP Server 或独立 Worker 形式存在**；  
  2. 小熊猫只做 MCP Client / 网关，不在 Next.js 进程中执行任意脚本；  
  3. 继续坚持「按意图只挂载当前需要的工具」「即用即删」的策略，让 5700+ 技能在用户体验上是「隐性丰富、显性克制」。
