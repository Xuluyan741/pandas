## 一、本次聊天逐句要点（按时间顺序概括）

1. **轻量 Agent 生态对比需求**
   - 用户给出 Nanobot / OpenClaw 语系的一批项目（NanoClaw、memU、PydanticAI、Claude Code、AnythingLLM、PicoClaw/TrustClaw），并说明自己偏好「极致性能 + 代码透明度」的 HKUDS 风格。
   - 希望将这些项目与小熊猫进行结构化对比，重点关注：轻量、Token 开销、动态技能/随用随下、MCP 支持等维度。
   - 用户提出「随用随下」的技术设想：探测 → 下载 → 动态挂载（如 Python importlib）→ 用后销毁，并询问是否可以用 Python 热加载/热卸载来实现。

2. **对比分析与技术建议阶段**
   - 助手整理了小熊猫、Nanobot、OpenClaw 的定位差异：小熊猫是「日程/任务垂直 + 轻量 Token + Web 优先」，Nanobot/OpenClaw 更偏「通用 CLI/多通道」。
   - 按三类项目做对比：极致轻量派（NanoClaw、memU）、现代架构/MCP 派（PydanticAI、Claude Code）、工具箱/桌面派（AnythingLLM、PicoClaw/TrustClaw），并分析小熊猫可以借鉴的点。
   - 给出「随用随下」的四阶段方案（探测 / 下载 / 挂载 / 销毁），指出小熊猫现有的 ClawHub 即用即删已经覆盖了「探测 + 挂载 + 销毁（描述级别）」，缺的是「可执行技能」层的能力，推荐用 MCP Server 作为技能单元。

3. **用户进一步需求：按意图只挂载部分 MCP 工具**
   - 用户明确提出要**设计并实现**「按意图只挂载部分 MCP 工具」的接口与调用流程（随用随下），要求真正落地在小熊猫项目中。
   - 目标：在每一轮对话中，只把和当前指令相关的 MCP 工具挂载到模型 `tools` 列表里，用完不保留，降低 Token 消耗。

4. **实现阶段（后端+AI 层）**
   - 新增意图筛选模块，用于从 MCP `tools/list` 的全量结果里选出本轮要挂载的工具子集。
   - 扩展统一 AI 调用层，使其支持 OpenAI/DeepSeek 风格的 `tools` 入参与 `tool_calls` 解析，并将执行结果以 `role: "tool"` 回填，再补全一轮。
   - 基于上述能力设计两个 HTTP API：一个只返回「本轮应挂载的工具列表」，一个直接完成「意图筛选 + 工具挂载 + MCP 执行 + 再补全」的对话流程。

5. **测试与文档整理**
   - 为意图筛选逻辑补充 Vitest 单元测试，验证匹配/排序/max 限制等行为。
   - 运行 `pnpm test` 确认现有测试通过，并新增测试绿灯。
   - 编写 `docs/MCP-按意图挂载.md`，将接口、流程与实现位置系统化记录，并在 `docs/对比-轻量Agent与随用随下.md` 中补充「小熊猫已实现：按意图只挂载 MCP 工具」章节。

## 二、关键结论（架构与设计层面）

- **1）小熊猫在轻量 Agent 生态中的位置**
  - 小熊猫：Next.js Web 应用，专注「日程/任务 + 冲突消解 + 长期目标」，拥有配额可视化、加权计数、奖励返还等节省 Token 的机制。
  - 与 Nanobot/OpenClaw 相比：不追求多通道/系统级控制，而是做「日程场景的垂直助手」，但在 Token 控制和技能扩展上延续了 HKUDS 的极简与可观测性思路。

- **2）「随用随下」的推荐实现形态**
  - 技能 = MCP Server 或其暴露的工具集合；不再用「传统插件」作为扩展单元。
  - 请求级别流程：探测（按意图发现需求）→ 只为本轮挂载需要的 MCP 工具 → LLM 补全（可能触发 tool_calls）→ 执行工具 → 回填工具结果 → 再补全 → 请求结束后逻辑上销毁（不在下一轮继续挂载）。

- **3）ClawHub 即用即删 + MCP 意图挂载的组合**
  - ClawHub：负责「技能描述级别」的随用随下——按用户输入搜索社区技能，只在当前轮的 system prompt 里注入相关 SKILL.md 片段，不持久化到全局。
  - MCP：负责「可执行能力级别」的随用随下——按意图从 MCP `tools/list` 中选出部分工具，只在当轮传给模型作为 `tools`，触发 tool_calls 时才真正执行。

- **4）Python 热加载仅作为 Worker 侧参考**
  - 对于 Nanobot 或自建 Python Worker，可以用 `importlib` + 删除模块的方式实现「下载脚本→执行→卸载」。
  - 小熊猫自身仍保持：Next.js/TypeScript 前端与 API，不在 Node 进程中随意执行下载来的脚本；可执行逻辑统一走 MCP Server 或独立 Worker。

## 三、新增/重要配置与接口一览

### 1. 环境变量（结合《功能测试清单》）

- **MCP 基础配置**（来自 `docs/功能测试清单.md`）：  
  - `MCP_SERVER_URL`：指向 HTTP MCP 服务地址，开启 MCP 能力的前提。  
  - `MCP_SERVER_HEADERS`：可选，JSON 格式 header，用于认证等。
- **社区技能 / ClawHub**：  
  - `CLAWHUB_API_BASE`：不为 `0` 即启用 ClawHub 即用即删；设为 `0` 可完全关闭社区技能搜索。
- 其他主功能相关变量（DeepSeek、搜索、登录、推送、数据库、支付等）详见：`docs/功能测试清单.md` 中「一、测试前准备」。

### 2. MCP 按意图挂载相关接口

- **（1）按意图只返回工具子集**
  - 路径：`GET /api/agent/mcp/tools/for-intent?text=<用户输入>&max=5`
  - 逻辑：
    - 从 `MCP_SERVER_URL` 指向的 MCP 服务器获取 `tools/list`。
    - 使用 `selectMcpToolsForIntent(text, tools, { maxTools })` 对工具按 name/description 与用户文本的关键词相关性排序。
    - 只返回得分大于 0 的前 `maxTools` 个工具；不修改全局配置，也不持久挂载。

- **（2）一步到位的 MCP 对话（挂载 + 执行）**
  - 路径：`POST /api/agent/mcp/chat`
  - 请求体：`{ "text": "用户指令" }`
  - 逻辑：
    1. 对 `text` 做意图分析，调用 MCP `tools/list` 拿全量工具。
    2. 用 `selectMcpToolsForIntent` 选出本轮要挂载的工具子集。
    3. 将这些工具映射为 DeepSeek 的 `tools` 定义，调用 `getUnifiedCompletionWithTools(messages, tools)`。
    4. 若模型返回 `tool_calls`：
       - 用 `mcpCallTool` 对每个 `tool_call` 在 MCP 侧执行；
       - 把结果作为 `role: "tool"` 消息携带 `tool_call_id` 回填到对话历史。
       - 再进行一次补全（最多 2 轮工具调用），最终返回文本回复与统计信息（挂载哪些工具、用了多少轮 tool_calls）。

### 3. 内部关键模块

- `src/lib/mcp-intent.ts`：  
  - `selectMcpToolsForIntent(userText, tools, { maxTools })`  
  - 行为：对用户文本做简单 token 化（小写、去标点、按空格分），根据 token 是否包含在工具的 name/description 中来打分，并按得分排序后截断。

- `src/lib/ai/types.ts`：  
  - 新增 `ToolDefinition`、`ToolCallItem`、`CompletionMessage`：  
    - `CompletionMessage` 支持四种角色：`system` / `user` / `assistant`（带可选 `tool_calls`）/ `tool`（带 `tool_call_id`）。

- `src/lib/ai/unified.ts`：  
  - 原有：`getUnifiedCompletion`（普通补全）。  
  - 新增：`getUnifiedCompletionWithTools(messages, tools)`：\n    - 会把 `CompletionMessage` 序列化为 DeepSeek 所需的消息格式；\n    - 支持将 `assistant.tool_calls` 序列化为 `tool_calls` 数组；\n    - 解析模型返回中的 `tool_calls`，映射为 `ToolCallItem[]` 返回给调用方。\n

- `src/app/api/agent/mcp/chat/route.ts`：  
  - 串联 MCP 客户端、意图筛选和 AI 补全，完成全流程。

- `src/lib/__tests__/mcp-intent.test.ts`：  
  - 针对 `selectMcpToolsForIntent` 的单元测试：空输入、空工具列表、命中 name/description、max 限制、无匹配等场景。

## 五、后续可选演进方向（供以后讨论）

- **1）更智能的意图与工具匹配**
  - 目前匹配基于简单 token 包含，可逐步引入：\n    - 基于 Embedding 的工具语义检索；\n    - 利用现有模型路由中的 Intent 分类结果（`ACTION` / `EXTRACTION` 等）来决定是否允许挂载高风险工具（如 shell）。\n

- **2）与 ClawHub 的更深度联动**
  - 将 MCP 工具与 ClawHub 技能做映射：当选中某个社区技能时，若其声明了 MCP 能力，则自动将对应 MCP 工具加入本轮挂载列表。

- **3）前端可视化与用户控制**
  - 在对话 UI 中展示「本轮挂载的 MCP 工具列表」，并允许用户手动关闭某些工具，以进一步控制 Token 与风险。
