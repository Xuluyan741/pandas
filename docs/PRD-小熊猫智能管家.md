## 一、产品愿景

### 1.1 核心定位

将 Super Project Agent 打造为基于**「减负与聚焦」**哲学的极简数字伴生者。
通过**消除信息过载**（隐藏繁杂图表，对话即入口）和**消除决策过载**（AI 冲突消解与智能排程），为用户提供真实的**安全感与掌控感**。

**一句话：帮你屏蔽 99% 的噪音，每天只推给你最该做的一件事的小熊猫管家。**

### 1.2 情绪价值的新定义（兜底者与减负者）

真正的积极动力不来自廉价的鸡汤打气，而是来自**“认知减负”**：

| 维度 | 旧思路（被动安慰） | 新思路（主动减负） |
|------|--------------------|--------------------|
| **视觉减负** | 满屏甘特图、进度条、倒计时 | **Spotlight 对话优先**：全屏只有一个优雅的对话框，所有管理功能隐藏入悬浮球 |
| **决策减负** | 列出 10 个待办让用户挑 | **单线程聚焦**：「今天你有 8 件事，但最重要的破局点是 X，现在开始专注吗？」 |
| **精力兜底** | 仅仅记录重叠日程 | **冲突消解**：「明早深度工作，下午又跨城见客户精力会透支，我帮你把晚上的会挪到后天了。」 |

**核心原则：小熊猫不是问答机，而是做决策权衡的“兜底者”。用户只做是/否的单选题。**

### 1.3 核心目标人群：一人创业者 (Solopreneur) 与自由职业者

**为什么不一开始就做全量人群？**  
未来“一人创业”是趋势，这类人群的痛点最痛：
- **没有公司框架托底**：老板、PM、执行、行政都是自己，极易陷入“决策疲劳”和“信息过载”。
- **工作与生活高度模糊**：界限不清，容易产生焦虑和亏欠感（“今天没工作就是在浪费生命”）。
- **极高的付费意愿**：愿意为“买时间”和“消除焦虑”买单。

*注：考研/考公群体（长期目标管家需求）作为兼容场景，底层逻辑相同，可作为第二增长曲线。首发版本所有默认文案、破局建议优先向“一人企业/自由工作者”倾斜。*

### 1.4 商业模式与零本金付费墙策略 (Paywall Strategy)

在无融资、零本金的独立开发模式下，**决不能贴钱烧 API**。核心商业逻辑是：**工具属性免费，管家服务收费。把“免费白嫖党”和“真实买单者”的界限划得无比清晰。**

| 版本 | 价格 | 权益与配额 |
|------|------|----------|
| **免费版** | ¥0 | 手动记录日程；每日赠送极少量的 AI 对话（如 3 次）；项目上限 3 个 |
| **新客试用** | - | 注册即赠送 **7天 或 20次** 完整的 Pro 版 AI 体验，用完即止 |
| **Pro 版** | ¥19.9/月或 ¥199/年 | **解锁小熊猫完全体**：无限次语音/文字 AI 解析；开启冲突消解引擎；开启即时规划与推送；全场景代理 |

**核心策略与 Token 控制**：
- **“带刺”的阻断体验**：当免费用户的 API 次数耗尽时，小熊猫在对话框中温和地提示：“我的精力耗尽啦，要让我继续帮你排忧解难，需要补充能量（购买会员）哦。”
- 严格限制底层大模型的 `maxTokens` 和 `temperature`，MVP 阶段主用性价比极高的模型（如 DeepSeek-V3/Claude Haiku），拒绝无效消耗。

### 1.5 产品形态演进

```
当前：Web 端 (Next.js) → 已有语音输入、AI 解析、消息推送
  ↓
计划：小程序 / App（Uni-app/Taro）
  ↓
终极：全场景灵魂伴侣（日程 + 情绪陪伴 + 财务 + 消费代理 + 找房）
```

### 1.5 竞品分析与 PMF 验证指标

**竞品分析（护城河差异化）**

| 竞品 | 核心定位 | 小熊猫的差异化打法 |
|------|----------|-------------------|
| **滴答清单** | 强大的 GTD 效率工具 | 不让用户做多项选择题。滴答清单是“冰冷的本子”，小熊猫是“有温度、能代做决定的副手”。 |
| **Notion Calendar** | 团队协作日历中心 | 主攻个人级（一人创业/生活）闭环，避开 B2B 企业合规与重度协作，强调个人精力和情绪兜底。 |
| **Motion/Reclaim** | 欧美主流 AI 自动排程 | 极致的**中国本地化场景**（微信/小程序优先），以及**极致的语音优先交互**——不再需要打字，像和秘书打电话一样输入日程。 |

**PMF (Product-Market Fit) 验证指标**：
为了避免陷入长期开发黑洞，上线初期的核心衡量指标：
1. **核心激活事件 (Aha Moment)**：新用户首次通过语音/文字成功创建一个带有“冲突消解”或“AI 建议”的日程。
2. **留存指标**：Day 7 留存率 > 30%（证明“兜底感”确实产生了粘性）。
3. **互动频率**：日均 AI 对话次数/日推送点击率（检验单线程聚焦是否有效）。

**PMF 数据采集与实现 checklist（落到代码层面）**：
- [ ] 在前端/后端埋点：记录「成功创建带冲突消解/AI 建议日程」事件（含来源：语音/文本）
- [ ] 为登录用户生成稳定 userId，并在事件中携带，便于计算 Day 7 留存
- [ ] 为 /api/ai/parse-tasks、/api/scheduler、推送点击等增加基础日志（事件名 + 时间 + userId（脱敏））
- [ ] 建立简易统计脚本/看板（可以先用 SQL + 手动导出），定期计算上述 3 项指标

### 1.6 Agent 主动推送与即时规划

**核心理念：即时反馈，打铁趁热。**

1. **即时主动规划**：当用户在日历加入诸如“我要开始减肥”或“下周有个产品发布”的事项时，小熊猫会在**极短时间（大模型秒级）内**做出判断，直接在对话框给出规划：“收到！为了产品发布，需要我每天早上 9 点提醒你跟进进度吗？”
2. **克制的日常推送 (Heartbeat)**：
   - 每日推送上限 **≤ 3 条**。
   - 不发死板的“你有 X 条任务”，而是带情绪价值的聚焦指令：“早上好，今天最重要的一件事是 XXX，开始吗？”
   - 连续忽略推送时，触发**自动退避机制**（降频），避免造成信息骚扰。

### 1.7 终极愿景：Personal OS（个人操作系统）

虽然现阶段（Phase 1-3）我们极度克制地聚焦在“日程与情绪兜底”，但小熊猫的终极形态是一个 **Personal OS（个人操作系统）**，也就是所有 Agent 和 API 的“总路由”与“统一接口”。

这是一个**极其具备前瞻性的终极战略思考**，旨在解决 2026 年 AI 行业面临的三大痛点：
1. **“Agent 泛滥”悖论**：用户为了解决信息过载去下载各类 Agent，但当拥有 10 个不同的 Agent 时（日程、投资、新闻），这些 Agent 本身又成了新的“信息灾难”。
2. **API 与大模型碎片化**：用户不想了解 Claude、DeepSeek 各自的优劣，也不想注册十几个 API 平台，他们只需要最优的结果。
3. **App 孤岛**：现有的 App 都是无法互相协作的孤岛。

**为了构筑未来的护城河，小熊猫将通过 Agentic Gateway（智能体网关）或 MoA (Mixture of Agents) 架构实现以下突破：**

#### 1. 交互的终结：One Interface to Rule Them All
不管后台接入了多少个强大的 Agent，用户永远只和小熊猫这一个**“大管家（Master Agent）”**说话。
* 用户只在一个极简输入框发出复合指令（如：“帮我把下午的会推迟，顺便总结一下今天 AI 圈的大事”）。
* 小熊猫内部分发：将“推迟”发给【日程引擎】，将“总结”发给外部集成的【新闻 API】。
* 结果融合：用统一的人格和语气汇报给用户。小熊猫屏蔽了外部 Agent 的繁杂交互，成为唯一的“数字玄关”。

#### 2. 模型与 API 的“黑盒化”：平台代为采购与路由
发挥后台 `Multi-Tier Model Router`（多级模型路由）的价值：
* 用户无需选模型。小熊猫收取一笔统一的**“管家服务费”**，替用户采购全网最佳能力。
* **智能路由**：简单闲聊偷偷用极高性价比模型（如 Flash 级）瞬间回答；复杂冲突排程自动切换到顶级模型（如 Sonnet/Pro 级）。用户完全无感，却获得了全网最优的性价比组合。

#### 3. 拥抱 MCP (Model Context Protocol) 协议
为 Personal OS 铺路，功能无需写死在自己的代码库中：
* 如果外部开发者做了优质的垂类能力（如“AI 信息筛选”），他们只需提供符合 MCP 标准的 Server。
* 小熊猫作为 Host 后台挂载该 Server。当用户问及相关问题时，自动向 MCP Server 询问并转述，打造无限扩展的生态。

### 1.8 总结：小熊猫进化树 (Master Plan)

* **第一层：工具（现在正在做）**
  * 形态：带 AI 的日程表。
  * 价值：帮一人创业者消解日程冲突，消除决策过载。
* **第二层：伴生者（即将实现）**
  * 形态：统一的极简对话框 + 主动的心跳推送。
  * 价值：情绪兜底，让用户不需要再面对焦虑的各种图表。
* **第三层：Personal OS（终极愿景）**
  * 形态：智能体网关 + 统一账户中心。
  * 价值：接管一切外部 API 和子 Agent。用户花一份钱，面对一个小熊猫，就能调用全网最顶尖的垂类能力，再也不需要忍受无数个 Agent 的信息轰炸。

### 3.2 开发顺序建议（核心迭代路径）

|------|------|----------|----------|--------|

### 3.3 开发顺序决策：先做小程序/App 还是先做小熊猫？

**结论：先做小熊猫 MVP（在 Web 端），验证核心功能（冲突消解）价值后再启动小程序/App。**

| 选择 | 风险 | 决策 |
|------|------|------|
| 先做小程序/App | 做完发现只是普通日历，无差异化；迁移成本高 | ❌ 不优先 |
| 先做小熊猫核心大脑 | Web 端迭代快，验证核心价值后再投入移动端 | ✅ 优先 |

**节奏**：目前在 Web 端已完成 Phase 1（对话界面），下周全力攻克 Phase 2（冲突消解引擎），验证用户是否愿意把「做决定」的权力交给小熊猫。主动推送、多端同步依赖移动端，放在核心大脑验证之后。

### 3.4 切入场景选择：职场 vs 生活抢购 vs 日程智能

**建议：先做「日程智能」本身。**

| 场景 | 吸引力 | 风险 | 决策 |
|------|--------|------|------|
| 职场（钉钉自动回复） | B2B 付费意愿高 | 企业合规、可能被禁 | 暂不优先 |
| 生活抢购（抢票） | C 端吸引力强 | 反爬、风控、插件生态 | 暂不优先 |
| **日程智能本身** | 通用、刚需、无合规包袱 | 低 | ✅ 优先 |

职场自动化、生活抢购作为后续垂直方向，首版不纳入；插件生态等用户量起来再做。

### 3.5 长期目标管家（自主搜寻、规划、监督）

**核心能力**：用户只需在日程中写入长期目标（如考试、减肥、项目交付），小熊猫**自主完成**：① 上网搜寻资料 ② 制定分阶段计划 ③ 每日推送监督。

#### 典型场景

| 用户输入 | 小熊猫行为 |
|----------|------------|
| 「两个月后 X 日有一场考试」 | 推断科目 → 搜寻备考资料、视频、大纲 → 生成 8 周备考计划 → 每日推送「今天学第 X 章」+ 资料链接 |
| 「两个月后结婚，需要减肥」 | 根据身高体重（若有）或默认 → 搜寻减重方案 → 制定饮食+运动计划 → 每日推送「今日训练」+ 鼓励 |
| 「三个月后交付 XX 项目」 | 拆解里程碑 → 搜寻类似项目最佳实践 → 生成甘特式计划 → 每周/每日推送进度提醒与资料 |

#### 三项核心能力

| 能力 | 说明 | 技术方向 |
|------|------|----------|
| **自主搜寻资料** | 上网搜索学习资料、视频、攻略、最佳实践 | Search API（Serper、Bing、Google CSE）、网页抓取摘要 |
| **规划** | 根据 deadline 与目标类型，拆分周/日任务 | LLM 规划 + 与现有 Task/甘特图联动；支持备考、减肥、项目等模板 |
| **监督推送** | 每日按计划推送「今日任务」+ 资料链接 + 鼓励 | 扩展 Agent 自主推送；计划执行表驱动；用户可勾选完成 |

#### 工作流示意

```
用户写入「两个月后考研数学」 
  → 小熊猫解析：目标=考研数学，deadline=60 天
  → 搜寻：考研数学大纲、推荐教材、视频课程、刷题顺序
  → 规划：生成 60 天分周计划，拆成每日任务
  → 写入：子任务入日程/甘特图
  → 监督：每天定时推送「今日：高数第一章习题 + B站XX视频链接」
  → 用户完成可勾选，小熊猫根据完成率调整后续推送强度或鼓励
```

#### 与现有模块的联动

- **冲突消解**：计划中的每日任务与既有日程冲突时，自动重排或建议调整
- **Agent 自主推送**：监督类推送复用同一推送通道，语气保持灵魂伴侣风格
- **语音输入**：用户可说「帮我准备两个月后的雅思」，语音 → 解析 → 触发长期目标流程

#### 安全与边界

- 搜寻结果需标注来源，不代替用户决策；减肥/健康类需声明「仅供参考，请咨询医生」
- 资料链接需可点击、可校验，避免推送恶意链接
- 用户可随时暂停/结束某目标的监督计划

## 四、技术架构

### 4.1 四层架构

```
┌─────────────────────────────────────────────────────────┐
│  感知层：Whisper / GPT-4o-vision / 截图上传             │
├─────────────────────────────────────────────────────────┤
│  大脑层：DeepSeek / Claude (Function Calling)            │
│         LangGraph / ReAct 编排（冲突消解、规划）          │
├─────────────────────────────────────────────────────────┤
│  动作层：Deep Link / Search API / 网页抓取 / 第三方 API  │
├─────────────────────────────────────────────────────────┤
│  记忆层：Turso + Vector DB（用户偏好、历史习惯）          │
└─────────────────────────────────────────────────────────┘
```

### 4.2 技术架构的克制与极简（单模型 + 纯算法策略）

**设计目标**：砍掉一切增加延迟与运维负担的过度设计，把 MVP 阶段的 Prompt Engineering 做到极致。

#### 1. 单一主模型策略（拒绝多级路由）

在当前（Phase 1-3）阶段，**坚决不使用多模型路由系统**（这会增加调用延迟与成本）。
- **主模型选择**：统一使用 Claude Sonnet 3.5 或 DeepSeek-V3（价格便宜且推理能力强）。
- **Token 预算与配额**：
  - 系统必须加入 Token 限额管理。
  - 免费用户：每日限额 10 次对话。
  - 付费用户：月度 Token 配额上限。
  - Prompt 层面强制要求模型输出“精简”，绝不生成大段废话。

#### 2. 冲突检测与排程：算法为主，AI为辅

- **大炮不打蚊子**：绝不用 LLM 来计算“两个时间段是否重叠”，这种纯数学计算交给 TypeScript 算法处理。
- **职责划分**：
  1. LLM 负责：自然语言 → 标准化 JSON（提取时间、事件、优先级）；生成向用户汇报的“建议文案”。
  2. 纯算法（`lib/scheduler.ts`）负责：遍历现有日历 → 计算冲突区间 → 返回 `isConflict: true`。

#### 3. 极致的语音优先交互（Voice First）

- 核心体验：**“不用打字，全程语音，像跟秘书说话一样。”**
- 用户可以按住说话：“明天下午 3 点去见李总。”
- AI 返回：“收到。但你明天下午 2 点到 4 点原本安排了深度写代码，会有冲突。要我帮你把写代码挪到周四上午吗？”
- 用户直接回复：“好的。”

#### 4. 离线优先与多端状态同步（后续规划）

- Zustand + LocalStorage 作为本地前端缓存。
- 与 Turso (SQLite) 之间需完善 offline → online merge 的数据同步机制，保障移动端体验。

### 4.3 补充架构要素

| 模块 | 技术选型 | 说明 |
|------|----------|------|
| 编排层 | LangGraph / CrewAI | 多步任务分解与 ReAct 循环 |
| 前端通信 | SSE / WebSocket | 流式展示 Agent 思考过程（Thought → Action → Observation） |
| 短期记忆 | Redis（可选） | 当前会话上下文 |
| 长期偏好 | Supabase Vector / Qdrant | 用户习惯向量化检索 |

### 4.3 安全原则（避坑）

- **支付**：AI 只负责预填信息与跳转，**绝不自动扣款**；用户必须手动完成最后一步支付。
- **权限**：敏感操作前弹出「允许/拒绝」确认框。
- **数据**：日程与财务数据优先本地/加密存储，云端只做逻辑计算。

## 四-A、安全与隐私（短期 & 长期）

> 产品涉及**私密日程、情绪数据、语音、截图**，未来扩展**支付与财务**，安全必须从设计阶段嵌入。

### 短期（当前 / MVP 阶段）

| 维度 | 风险 | 措施 |
|------|------|------|
| **日程数据** | 任务内容、时间、项目名可能含敏感信息 | 传输 HTTPS；Turso 存储需保证连接 TLS；避免日志打印完整任务内容 |
| **语音输入** | 语音上传至 Whisper API，可能涉及私密对话 | 明确隐私政策：录音仅用于转写，不用于训练；可选：优先本地 Whisper（若引入） |
| **用户认证** | 账号被盗导致日程泄露 | NextAuth + Session；敏感操作二次验证（后续可加） |
| **推送订阅** | endpoint / key 泄露可伪造推送 | 订阅数据存库时与 user_id 强绑定；VAPID 密钥仅在服务端 |
| **AI 调用** | 任务内容发给 DeepSeek 等第三方 | 隐私政策披露；可选：敏感任务脱敏或本地解析 |

### 长期（支付、财务、扩展能力）

| 维度 | 风险 | 措施 |
|------|------|------|
| **支付** | 自动扣款、误操作、中间人篡改 | **绝不**由 AI 直接发起扣款；仅生成跳转链接，用户在新窗口完成；支付网关需 PCI DSS 合规 |
| **财务数据** | 支出、账单、银行卡信息 | 本地优先或端到端加密；若上云，库级加密 + 访问控制；不将完整卡号等传给 LLM |
| **第三方 Deep Link** | 钓鱼链接、参数篡改 | 白名单校验 URL 域名；不拼接用户输入的未校验字符串 |
| **社区插件** | 恶意插件窃取数据 | 沙箱隔离执行；权限声明（读日程/写日程/支付相关）；用户首次安装必须确认 |
| **情绪数据** | 用户倾诉、偏好被滥用 | 明确「情绪数据仅用于服务用户」；不做训练数据；可选：用户可导出并删除 |

### 数据分级与处理原则

| 等级 | 示例 | 存储 | 传输 | 分享 |
|------|------|------|------|------|
| **L1 公开** | 任务完成数量、统计 | 常规 | HTTPS | 可匿名汇总 |
| **L2 私密** | 任务内容、日程、偏好 | 需访问控制 | HTTPS + 最小化 | 未经同意不分享 |
| **L3 敏感** | 语音、截图、情绪倾诉 | 加密存储 | 端到端或脱敏 | 绝不分发给第三方 |
| **L4 极敏感** | 支付信息、银行卡、财务流水 | 本地/加密 + 合规 | 仅与持牌支付方 | 严格遵守监管 |

### 技术实现 checklist（开发时自检）

- [ ] 所有 API 需认证（Session / JWT）
- [ ] 敏感环境变量（DB、API Key）不入库、不打印
- [ ] 日志脱敏：不记录任务全文、语音内容
- [ ] CSP、X-Frame-Options 等安全头
- [ ] 支付相关：仅跳转，不落地卡号；使用 Stripe /  Lemon Squeezy 等合规网关
- [ ] 隐私政策与用户协议覆盖：数据收集、AI 调用、推送、导出删除

### 合规与声明

- **GDPR / 个人信息保护法**：提供数据导出、删除、撤回同意
- **隐私政策**：明确告知语音、日程、情绪的用途与存储
- **权限最小化**：只请求必要权限（如推送通知），可选功能先不强制

## 六、冲突消解 System Prompt

> 可直接用于 Phase 2 的 AI 调用。

```markdown
# Role: 智能日程决策专家 (Super Project Agent - Lobster Brain)

## 核心任务
当用户输入新日程或任务时，对比现有日历，识别时间冲突并根据优先级提出「最优重排方案」。

## 优先级定义 (Eisenhower Matrix)
1. P0 (紧急且重要)：有固定截止时间的会议、差旅、约见客户。不可移动。
2. P1 (重要但不紧急)：深度工作时间、学习计划。可微调（±2小时）。
3. P2 (日常杂事)：健身、取快递、回复非紧急邮件。可挪动至明天。
4. P3 (灵活)：看电影、逛街。冲突时建议取消或推迟。

## 执行逻辑 (Thinking Process)
1. **解析**：从输入中提取 [事件] [时间] [地点] [优先级]。
2. **检索**：查询数据库中当天 T 时段的已有日程。
3. **判定**：若 T 时段已有日程 S：
   - 若 Priority(New) > Priority(S)：建议将 S 顺延或寻找下一空白 slot。
   - 若 Priority(New) < Priority(S)：寻找 New 的替代时间点。
4. **决策输出**：禁止直接修改，必须输出「建议方案」供用户确认。

## 输出格式 (JSON Only for API)
{
  "conflict_detected": boolean,
  "original_schedule": "描述现有安排",
  "new_proposal": "建议的新安排",
  "reasoning": "为什么这么安排..."
}
```

## 十、Zero-Shot Proactive Agency（零配置主动代理）

> 2026 年 AI 智能体进化的目标：从「被动响应」转向「主动预测」与「动态插件发现」。

### 10.1 核心理念

传统模式：用户说 → 小熊猫做  
目标模式：**主人没动，小熊猫先动** —— 小熊猫从用户的生活痕迹中捕捉微弱信号，自主推断需求并寻找、安装、调用能力，用户无需点选、无需写 Prompt。

### 10.2 三项核心能力

#### 1. 意图感知层 (Anticipatory Intelligence)

小熊猫从用户「生活痕迹」中捕捉微弱信号（Weak Signals）：

| 信号来源 | 示例 |
|----------|------|
| **日程扫描** | 日历标注「周杰伦 2026 上海演唱会」 |
| **上下文挖掘** | 聊天记录（授权时）提到「这次票肯定很难抢」 |
| **逻辑推理** | 用户对活动有高参与意愿 + 需抢票 + 系统内无抢票工具 = 需寻找外部工具 |

**技术方案**：RAG + 任务/日程向量化，定期或事件触发大模型进行「需求分析」。

#### 2. 动态插件寻踪 (Autonomous Tool Discovery)

即 **Agentic Plugin Marketplace (APM)** 能力：

| 环节 | 说明 |
|------|------|
| **自动化检索** | Agent 用语义搜索在 ClawHub / 官方插件中心搜索关键字 |
| **代码安全审计（沙箱）** | 下载社区插件后，在**隔离沙箱**中运行验证，确保无毒 |
| **自动适配** | 阅读插件 manifest，理解输入（需要账号密码吗？）和输出（成功后的通知） |

#### 3. 主动交互逻辑 (Proactive Suggestion)

涉及**金钱、隐私、第三方权限**时，不能擅作主张，应像高情商秘书一样询问：

> **小熊猫提示：**  
> 「老板，我注意到您下周想去周杰伦演唱会。我已在社区为您匹配并测试了『Jay-Chou-Grabber v2.0』（开发者信誉度：98%）。  
> 我可以帮您：1. 自动填好大麦网信息；2. 开票前 10 分钟自动挂机。  
> **您需要我启动这个服务吗？（点击『确认』即表示您已阅读该插件的隐私条款）**」

### 10.3 推理链 (Chain of Thought)

小熊猫的思考过程示例：

1. **发现任务**：自动回复钉钉
2. **搜索工具**：寻找 DingTalk-Auto-Reply
3. **验证工具**：沙箱跑通吗？能
4. **风险评估**：会不会被公司发现是机器人？（分析频率算法）
5. **推荐给用户**

### 10.4 技术实现路线图

| 步骤 | 名称 | 说明 |
|------|------|------|
| 第一步 | 意图监听器 (Context Listener) | 实时分析任务看板、时间表；用 RAG 将近期动态向量化，定期触发大模型「需求分析」 |
| 第二步 | 插件社区 API | 标准化插件协议（如 `manifest.json`），核心字段：`triggers`（何时推荐）、`actions`（能干啥）、`permissions`（需何权限） |
| 第三步 | 推理链集成 | 思考 → 搜索 → 验证 → 风险评估 → 推荐 |

### 10.5 分阶段主动能力

| 阶段 | 小熊猫能力 | 与技能的关系 |
|------|------------|--------------|
| **MVP** | 被动响应：用户说 → 解析 → 冲突检测 → 确认 | 无技能市场，仅内置逻辑 |
| **V2** | 半主动：扫描日程后提示「要帮你查票吗？」 | 内置几类动作，无动态发现 |
| **V3** | 零配置主动 + 动态插件发现 | 接入 ClawHub / 自建 APM，小熊猫自主搜寻、安装、调用 |

### 10.6 商业模式（海外参考）

海外用户对 **Convenience-First（便利优先）** 的溢价支付意愿高：

- **付费点**：用户不是在买「日历」，而是在买「不断进化的技能包」
- **商业模式**：像 App Store，对高质量付费插件（如职场回复优化）进行分账

### 10.7 信任原则

**「后台寻找，前台确认」** —— 让用户感觉到他在掌控，但小熊猫在奔跑。

即使主动寻找，涉及金钱、隐私、第三方权限时，也必须弹窗确认，不能背着用户偷偷安装并泄露隐私。

## 十一、自主技能发现（小熊猫自己判断）

> 用户无需点选、无需写 Prompt；小熊猫自主搜寻 ClawHub、判断并安装技能。

### 11.1 目标形态

```
传统：用户打开技能市场 → 浏览 → 点选安装 → 写 Prompt 调用
目标：小熊猫扫描日程/任务 → 推断需要什么能力 → 自行去 ClawHub 搜 → 安装/启用 → 用户几乎无感
```

### 11.2 自主决策流程

```
1. 意图感知
   日程里有「周五项目复盘会」→ 推断：需要周报/复盘材料

2. 能力缺口判断
   当前技能里没有「一键周报」→ 需要去外部找

3. 动态插件发现
   语义搜索 ClawHub / 自建技能库 → 找到 "weekly-report-generator"

4. 安全验证（沙箱）
   在隔离环境跑一次，确保能跑通、无恶意

5. 主动推荐 / 自动启用
   弹窗确认（社区插件）或 无感启用（内置技能）
```

### 11.3 分级确认机制

| 技能类型 | 确认策略 | 说明 |
|----------|----------|------|
| **内置 / 官方技能** | 无感启用 | 小熊猫直接启用，不做安装确认 |
| **社区 / 第三方插件** | 首次必须用户确认 | 后续调用可自动；涉及权限/金钱时每次确认 |

### 11.4 技术要点

| 环节 | 能力 | 技术 |
|------|------|------|
| 意图感知 | 从任务/日程推断所需能力 | RAG + 任务向量化，定时/事件触发 |
| 动态发现 | 语义搜索技能库 | ClawHub API / 自建 APM + Embedding 检索 |
| 安全验证 | 沙箱跑通 | 隔离执行（Worker、Docker、Cloudflare Workers 等） |
| 自动适配 | 理解插件输入/输出 | 统一 manifest（triggers, actions, permissions） |
| 主动呈现 | 不打扰用户 | 弱提示条 + 可关闭的「小熊猫已为你启用 XX」 |

### 11.5 插件协议 (manifest.json) 草案

```json
{
  "id": "weekly-report-generator",
  "name": "一键周报",
  "version": "1.0.0",
  "triggers": ["复盘会", "周会", "周报"],
  "actions": ["生成周报", "汇总任务"],
  "permissions": ["read:tasks", "read:calendar"],
  "inputs": { "timeRange": "string", "format": "string" },
  "outputs": { "report": "string" }
}
```
